## Scope (rotate across these; don't over-fit one area)

### 1. Spells — at least 6 new ones across schools

Pick from Diablo-shaped effects, but adapt them to fit what's already in the codebase:
- **Damage shapes:** chain lightning (jumps), frost nova (ring AoE), blizzard (timed-zone), hydra (turret), meteor (delayed impact), frozen orb (travels then bursts), charged bolt (multi-projectile spread), blood star (single-target piercing).
- **Utility / control:** teleport (positional), slow time (zone debuff), nova-style cleanse.
- Each spell gets full test coverage: damage curves vs. tier, status effects, AoE shape, cast cost, cooldown, line-of-sight if applicable.

### 2. Gear affixes — D2-style prefix/suffix system

- Roll tiers from item level (iLvl -> eligible affix pool, weighted by rarity).
- Affix categories: stat bumps (+strength, +life regen, +mana, +X% resist), proc affixes (chance on hit to cast Y), conditional (only-vs-undead, +damage to demons), defensive (X% physical reduction, dodge, +life on kill).
- Affix DB data-driven; build-time integrity check (same shape as the content-integrity suite the previous loop added).
- Mods stack with sensible cap rules.

### 3. Gear that grants spells

- "+N to Fireball" tier bonuses.
- Weapon-bound spells ("casts X on hit", "casts Y on crit", "casts Z when struck").
- Charges system (limited-use spell on item).
- Must compose with the existing spell system, not special-case. Tests cover the spell-grant resolution path explicitly: same spell from item vs. learned must hit the same damage/effect code path.

### 4. Cast & self buffs

- **Targeted buff casts:** party/single-target enhancement (e.g., haste, shield, regen).
- **Self-buffs:** shouts, stances, auras.
- Buff data model uniform with existing status-effect system if at all possible — don't fork.
- Duration system + refresh-vs-stack rules + dispel/cleanse interactions.
- Tests must cover: buff stacks correctly, refreshes don't double-extend, dispels strip the right ones, conflicting auras resolve deterministically.

### 5. Rendering — Diablo capability tier

The 2026-06-13 loop's "Needs a human" section flagged visual work as un-loopable without a human eye. Cover the *data and logic* side here; document anything genuinely needing a human in the next SUMMARY.

- Spell impact effects: data + system, not pixels (e.g., a `frostCloud` effect descriptor + lifetime + tick logic the renderer can later draw).
- Damage numbers + hit-flash: data path + tests; render hook can stay a thin shell.
- AoE ground decals: position + radius + duration data, plus the lookup the renderer consumes.
- Basic particle pooling: cap N active particles per type, recycle pool. Tests for pool exhaustion + recycle correctness.
- Buff icons over portrait + over character: the data side (which buffs are visible, their icon refs, sort order) — gate the actual rendering behind sane unit tests.
